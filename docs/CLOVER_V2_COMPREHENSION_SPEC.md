# Clover Comprehension Reliability Standard & Benchmark Suite (v2)

> **Status:** the measurable quality standard for Phase 3 comprehension and the gate that must clear before zero-touch auto-run. Last updated 2026-06-20.
> Companion to [CLOVER_V2_ROADMAP.md](CLOVER_V2_ROADMAP.md) and [CLOVER_V2_PHASE3_SPEC.md](CLOVER_V2_PHASE3_SPEC.md).


## 1. Purpose & the No-Miss Product Promise

Clover's deliverable is not "a good summary." It is a **trustworthy, no-miss action list** an operator's tenants can run their work from without re-reading the source. The single failure that destroys that promise is the **silent miss**: a real action item, reference number, deadline, party, or amount that exists in the thread or its attachments but never reaches the output, with nothing flagging its absence. A loud failure (flagged `needs_review`) is acceptable; a silent one is the cardinal sin because the user cannot know to compensate.

This standard makes silent misses **measurable, gated, and regression-tracked**. Reliability is architectural and measured — not a model you pick. The benchmark suite is the proof object: flipping the pipeline from manual to zero-touch auto-run is **gated** on clearing the bars in §5 on operator hardware.

**Honest foundation (changed in v2).** The previous draft rested the no-miss guarantee on a "deterministic ground-truth oracle" that does not exist in the codebase. The current pipeline (`clover/comprehend.py`) derives all facts from the AI via `_distill_prompt` / `_DISTILL_SCHEMA`; `_verify_facts` (lines 297–336) only confirms that AI-emitted values appear in the source — it never independently extracts anything, and `clover/rules.py` is classification-only. There is no RFI/NCR/date/amount extractor anywhere. This standard therefore distinguishes three tiers of evidence, and never claims more than the tier supports:

- **T1 — Deterministic recall floor (to be built).** A regex/parser extractor (`clover/eval/extractors.py`, §6) for the *catchable* classes only: `ref_number`, `date`, `amount`, `party` (the last from message headers). It is a **floor and a regression seed**, not ground truth: it can prove the AI dropped something the regex caught, but it **cannot prove completeness**. Its own recall must be characterized against the gold set (§4) before any bar depends on it.
- **T2 — Gold required-set (human-labeled).** The semantic classes — above all `action_item` — have **no deterministic oracle** and are gold-label-only. Recall for these is measured against human annotation, with the realistic ceiling being inter-annotator agreement, not 100% (§4).
- **T3 — Adversarial audit (escape detection).** The deepest silent miss is an atom present in the thread but caught by *neither* T1 nor the T2 annotators. A periodic red-team audit (§4) measures the **oracle escape rate** so this class is no longer invisible.

Two scoring asymmetries govern everything below:
- **Recall is the gate; precision is a secondary penalty.** A missed action item fails a thread outright. A spurious (hallucinated) item is bad but recoverable by one-click dismiss.
- **Headline metric is AND-gated per thread, never an average.** A 95% per-item average hides catastrophic single-thread drops. The headline is *% of threads with ZERO misses against the frozen gold required-set*.

> consolidated-action-list layer, the validate-the-validator (false-clear) layer, and the staged real-case

## 1.5 The Design Target: the Consolidated Action List is the Unit of Truth

**Objective (restated as the design target).** Clover reads **scattered communication** — email today; WhatsApp and other channels later — and converts it into a **perfectly itemized, cross-referenced to-do list**. The thing the standard optimizes is **not** a per-thread comprehension record; it is the **consolidated action list**, and every bar, gold artifact, and gate below is judged against that list.

- **Perfectly itemized** means, on the consolidated list: (i) **no action missed** (recall / no-miss at the *list* level); (ii) **no action duplicated or fragmented** (dedup/consolidation); (iii) each action **correctly attributed** — who owes what to whom, by when (attribution); (iv) each action **correctly statused** — open vs done/superseded/cancelled (status-resolution).
- **Cross-referenced** means each consolidated action links to **all** its source message(s) **across channels** AND to its **parent issue**, every link is **resolvable to the originating occurrence**, and the **same real-world action mentioned across multiple messages/threads/channels collapses to ONE consolidated item**.
- **Source-agnostic** means the list is invariant to *which channel* an action arrived on; adding a channel must never silently regress an existing channel.

**The list, not the thread, is the unit the user acts on, so the list is the unit of pass/fail.** A user never acts "per thread"; they run their work off one merged list. Therefore the cardinal gate moves from per-thread no-miss (§5 bar 1, retained as a floor) **up to** list-level no-miss (§9 CLOVER-LIST-RECALL). The evaluation unit is the **CASE** = one real-world situation spanning *N* messages across *M* channels, not the thread; gold is annotated and scored per CASE.

**Why this is a new layer, not an edit to §1–§5.** The existing suite scores an `action_item` *atom* per thread (and the pipeline does not yet emit even that — `_DISTILL_SCHEMA` in `comprehend.py` stops at `facts/contacts/tags`). The consolidated list adds failure modes the per-thread architecture **cannot exhibit**: cross-thread/cross-channel dedup, status resolved across the whole chronology, attribution that survives a reassignment, and cross-reference completeness across mentions. These get their own bars (§9), their own CASE-level gold (§10), and a fidelity layer (§11) that validates the whole instrument against reality before any model change.

**Prerequisite artifacts (cross-referenced to the existing build list, §6).** The list-level bars score an artifact the pipeline does not emit today. Until it ships they are **REPORTED-NOT-GATING** under the existing schema-prerequisite-debt rule (same posture as CLOVER-ATTACH in §3). The required artifacts are:

- **An `action` object** added to the emitted schema (extends `_DISTILL_SCHEMA`):
  `{action_id, text, owner, counterparty, direction, due_raw, due_canonical, status ∈ {open, done, superseded, cancelled, blocked, needs_review}, is_mine, source:[{channel, message_id, char_span, is_quote:bool, is_origin:bool}], parent_issue_id, consolidated_from:[message_ids across channels], coverage_flags:[...]}`.
- **A consolidation pass** (action clustering across threads/channels) — distinct from `projects.py`, which is a `facts.project` *string* group-by, not an action clusterer.
- **De-quoting, attachment-extraction, and table-structure preservation** — see §6 additions; these gate whether the list-level bars may CLEAR a corpus at all (§9.7).

## 2. Reliability Architecture as Design Requirements

Every comprehension stage MUST implement all six. These are requirements, not options.

**(a) Deterministic recall-floor extraction + AI cross-check.** `ref_number`, `date`, `amount`, and `party` (from headers) are extracted by deterministic rules and used to **cross-check** AI output: any T1 atom absent from AI output is a **hard miss** — force-injected and flagged. This is the biggest single lever against silent misses (the deterministic-first principle), but it is a **floor, not ground truth**: it cannot speak to anything it cannot catch. **`action_item` is explicitly out of T1's scope** — action items are semantic and not regex-able; the only deterministic signal here is a *candidate* surface ("please", "by <date>", "kindly", imperative verbs) that is **recall-suggestive, never authoritative**, and is logged as a hint for annotators, never counted as a caught atom.

**(b) Decomposed single-purpose passes.** No mega-prompt. Separate narrow passes (classify; extract facts; extract action items; chronology; summarize) so each is individually reliable and individually scorable. A failure localizes to a pass.

**(c) Recall-first action items.** Over-capture with one-click dismiss beats silent drop. The action-item pass is tuned for recall; precision is cleaned downstream.

**(d) Completeness-critic pass + coverage accounting.** A second pass asks "what did pass 1 miss?" against the T1 floor and the raw text. The critic is itself an AI pass and **will also miss silently** — so its recall is measured directly (§3 CLOVER-NOMISS reports a `critic_recall` sub-metric: of atoms pass 1 dropped that gold contains, how many the critic recovered) and it is never treated as a completeness guarantee. **Coverage accounting is mandatory and must be a real emitted field** (it does not exist today): every comprehension record MUST carry `coverage = {messages_total, messages_read, render_failed, attachments_total, attachments_inventoried, attachments_read, bytes_sent, bytes_truncated}`. Today's `_thread_messages` (comprehend.py lines 70–71) silently `continue`s on a render exception, dropping the message with no count — this is the exact "1/68" truncation class. The fix is mandated in §6: increment `render_failed` on that except, record `bytes_truncated` at every truncation point (the 200 KB per-message cap in `estimate_thread_tokens`, the `max_chars` limit in `_build_once`), and treat `messages_read < messages_total` OR `render_failed > 0` OR `bytes_truncated > 0` as a **loud failure**.

**(e) Source citation + verification.** Every emitted item (action, fact, summary claim) carries a `source = {message_id, char_span | attachment_id+page}`. Uncited items are flagged. Citations are **resolvable** by a deterministic verifier (the cited span must exist and contain the claimed atom). Trust = traceability (verify in 2 s), not a confident sentence.

**(f) Gold-set benchmark as the proof.** Don't trust, measure. The gold set (§4) + scorer (§6) produce the numbers in §5. No mass-run of the backlog until the bars clear; monitor continuously after.

## 3. The Clover Benchmark Suite

The suite is **staged to what the pipeline can actually measure today**, to avoid scoring features that do not exist. Building TEDS/CDM scorers for attachments the pipeline cannot yet read, or 1M-token brackets for synthesis that is not built, is speculative tooling.

- **Gating now (per-thread reliability, all features exist or are in §6 build list):** CLOVER-NOMISS, CLOVER-COVERAGE, CLOVER-IFADHERE.
- **Gating when its feature lands:** CLOVER-ATTACH (gates only once attachment ingestion exists — see §6 work item), CLOVER-TRIAGE (classification exists today, gates now), CLOVER-LONGCTX (benchmarks the *composite chunked path*, gates now).
- **Future suites, reported-not-gating until their feature is built:** CLOVER-CHRONO (full state-resolution scoring) and CLOVER-SYNTH (Phase-4 cross-thread issues, not yet built — see the roadmap). These are specified so the harness is forward-compatible, but they do not block present-day per-thread ship.

All free-text matching uses a **frozen matcher per `gold_version`** (§4): a named embedding model + a fixed cosine threshold for semantic entailment, with an "ignore" tag for genuinely-optional atoms. `ref_number` uses exact match after case/whitespace normalization. **`date` and `amount` use a canonical normalizer, not raw exact match** — both surfaces are parsed to a canonical value (date → ISO `YYYY-MM-DD`; amount → `{currency, decimal}`) and compared canonically, so `14 Mar 2025` / `14 March 2025` / `2025-03-14` collapse to one and neither undercount (exact) nor overcount (edit-distance) applies.

### CLOVER-NOMISS — No-Miss Recall (headline)
- **Tests:** recall of every required atom per thread, with an itemized miss list.
- **Construction:** gold required-set per thread = union of (T1 floor atoms) + (T2 human-labeled atoms, double-annotated per §4). Atom classes: `action_item, ref_number, party, date, amount, attachment`.
- **Scoring:** per class, `recall_c = covered_c / required_c`. An atom is *covered* iff an output item entails it under the frozen matcher; **ACE split is mandatory** — separately report `found_at_all_c` (atom located) and `value_correct_c` (located AND value/attribution correct), so a fuzzy match that flags a wrong/misattributed item as "covered" is exposed as a value-level miss rather than a recall success. **Headline `NoMissRate = (# threads where value_correct == required across ALL classes) / (# threads)`** — AND-gated per thread. Sub-metrics: `critic_recall` (§2d) and the literal uncovered-basis list per failing thread.
- **Reproducibility:** the matcher, threshold, and normalizer are pinned to `gold_version`; scores are comparable only within a version. For `action_item` the headline target is `recall ≥ measured inter-annotator agreement` (not a literal 100% — see §5 bar 1), with every gap loud-flagged.
- **Range:** each `recall_c`, `NoMissRate` ∈ [0,100]%.

### CLOVER-COVERAGE — Read-Completeness / Anti-Truncation
- **Tests:** that every message and every attachment was provably ingested — no silent drop or truncation.
- **Construction:** from the emitted `coverage` field (§2d) against the thread's actual member count and `_real_attachments` inventory (derivable today from `threads.py` `render_message` / `has_attach`).
- **Scoring:** `CoverageClean = (# threads with messages_read==messages_total AND render_failed==0 AND bytes_truncated==0 AND attachments_inventoried==attachments_total) / N`. Attachment *content* read is scored only once ingestion exists (see CLOVER-ATTACH gate condition).
- **Range:** [0,100]%.

### CLOVER-IFADHERE — Format & Resolvable-Citation Adherence
- **Tests:** machine-checkable output-contract assertions: valid JSON schema; every action item has owner + due-date fields; **every emitted item carries a valid resolvable source citation** (the cited span exists and contains the atom); chronology field is sorted; no message omitted from timeline; tags drawn only from vocab.
- **Construction:** each assertion = a pure verifier function returning pass/fail; no LLM judge.
- **Scoring:** instruction-level pass rate and **thread-level strict** `ThreadStrict = (# threads where ALL assertions pass)/N`. (The IFEval loose/8-variant cosmetic-wrapper analysis is **dropped** — it exists to test natural-language instruction following; Clover's output is machine-generated JSON against a fixed internal schema we control, so strict pass/fail is sufficient.)
- **Range:** [0,100]%.

### CLOVER-TRIAGE — Triage/Classification
- **Tests:** correct domain/category (and faceted tags) vs the operator profile.
- **Construction:** gold label per thread from operator review; faceted tags validated against profile vocab.
- **Scoring:** overall accuracy + **per-class recall** (macro and per-category) + tag-set F1, with a confusion matrix. Per-class recall surfaces a minority class silently collapsing.
- **Range:** [0,100]%.

### CLOVER-LONGCTX — Composite-Path Long-Context Retrieval & Reliable Ceiling
- **Tests:** whether the **deployed composite path** (raw window for short threads, the existing chunk/iterative-refine path in `_build_once`/`_refine` for long ones) holds recall as thread length grows — NOT the raw model window in isolation. This is the key v2 correction: a local model that fits 128 GB will not hold 100% multi-needle recall to 128k tokens *in a single window*, so the bar is set against the composite path the product actually runs.
- **Construction:** synthetic threads at lengths {4K, 8K, 16K, 32K, 64K, 128K} tokens with planted single-needle, multi-needle (return ALL), and multi-hop chains; vary needle **position** (early/middle/late) and density; routed through the real pipeline (chunking engages above `max_chars`).
- **Scoring:** per-length, per-position, per-density recall (multi-target = all required items for full credit). **`EffectiveThreadCeiling` = the largest length at which composite-path recall on `action_item`/`ref_number`/`party` stays at the no-miss floor.** Report the full length×position×density grid (a flat average hides "fact at msg 40/60 missed 70%").
- **Range:** [0,100]% per cell; ceiling in tokens.

### CLOVER-ATTACH — Attachment Comprehension *(gates only once attachment ingestion exists)*
- **Gate condition:** today `_block_text` reads body only and never reads attachment content, so attachment-content recall is currently scoring nonexistent functionality. CLOVER-ATTACH is **reported as 0/blocked** and does NOT gate until the §6 "attachment ingestion" work item ships. Attachment *inventory* (count, names) IS measurable today via `render_message` and feeds CLOVER-COVERAGE.
- **Tests (when live):** PDF text extraction, annotation/comment-layer extraction, table structure, drawing markup.
- **Scoring (when live):** text = `1 − NormalizedEditDistance`; tables = TEDS; annotations/fields = field-level F1. **Attachment atoms feed back into CLOVER-NOMISS** (a missed annotation deadline is a thread miss).
- **Range:** NED ∈ [0,1] (lower better); TEDS/F1 ∈ [0,100].

### CLOVER-CHRONO — Chronology *(reported, not gating until Phase 4)*
- **Tests:** chronological ordering of events and resolution of re-bound state (final value of a fact that was approved → superseded → re-approved).
- **Scoring:** ordering via `1 − NormalizedEditDistance` over the event sequence; state-resolution accuracy = fraction of re-bound facts whose final value is correct. Reported per run; not a gate bar until Phase-4 sequencing.
- **Range:** [0,100]%.

### CLOVER-SYNTH — Cross-Thread Synthesis *(future; gates Phase-4 zero-touch only, and only at hardware-justified brackets)*
- **Tests:** Phase-4 cross-thread issue clustering and the per-project no-miss action list — including **per-thread → project propagation**: an atom present per-thread but dropped during clustering is a synthesis miss, traced explicitly.
- **Construction & hardware note:** context brackets are **right-sized to the operator box**. The 128 GB local box runs ~128k-class context for per-thread work; 1M-context is only relevant to whole-project synthesis and is served by *separate* long-context backends. Brackets above what the local box can run are benchmark-only references and are **not** part of the ship gate unless a backend that actually holds them on operator-controlled hardware is identified. The previous draft's 256K/512K/1M local brackets are cut as a ship gate; SYNTH gates at the largest bracket the deployed Phase-4 backend demonstrably sustains.
- **Scoring:** issue-clustering pairwise F1 / cluster purity; issue-status accuracy; cross-project action-list `NoMissRate` (AND-gated). Slice every metric by bracket.
- **Range:** [0,100]% per bracket.

## 4. Gold Set: Construction Protocol

- **Bootstrap (deterministic candidate → human review):** run the T1 extractor (once built) to propose every catchable atom; annotators confirm/edit and **add the semantic atoms (action items) the rules cannot catch**.
- **Double annotation for semantic classes (new in v2):** the `action_item` required-set is built **independently by two annotators**; disagreements are adjudicated by a third. The suite reports **inter-annotator recall** as the *realistic ceiling* for action-item no-miss — a 100% bar against a single fallible annotator is theater, because the annotator can also miss. T1 classes (refs/dates/amounts/parties) may be single-annotated since the extractor anchors them.
- **Adversarial / red-team subset (new in v2 — closes the core silent-miss hole):** a small rotating subset (5–10 threads) is **exhaustively hand-extracted from raw text by a second auditor with no extractor priming and no sight of the bootstrap candidates**, specifically to find atoms that *both* T1 and the T2 annotators missed. Track **`oracle_escape_rate` = (atoms found by the auditor that the bootstrap pipeline did not surface) / (total atoms the auditor found)** as a first-class regression metric. Without this, `NoMissRate = 99%` can coexist with a far higher real-world miss rate, because gold would only measure "recall against what we already knew to look for."
- **Frozen matcher per version:** `gold.jsonl` pins the entailment matcher id, the cosine threshold, and the date/amount normalizer version. Changing any of them bumps `gold_version`; scores compare only within a version.
- **Minimum size before any bar is declared cleared (new in v2):** at least **N = 100 annotated threads** before NoMissRate or any recall bar is treated as cleared. A 99% rate on 30 threads means "fewer than one thread failed," statistically indistinguishable from a fluke. Start at 30 representative threads for development; do not ship-gate on them. Plus ≥10 attachment golds (used only when CLOVER-ATTACH is live) and ≥3 whole-project bundles (used only when CLOVER-SYNTH is live).
- **Growth rule:** every production silent miss, every operator `Resolve` correction, and **every red-team escape** is promoted into the gold set (regression seed). Grow until per-class recall variance stabilizes; target ~100–150 threads.
- **Representativeness (stratified):** sample across domain/category, thread-length buckets, attachment-heavy vs none, scanned-PDF vs digital, forwarded/quoted clutter, and re-binding/multi-hop cases. Mirror the real archive distribution (e.g. the heavy duplicate-link / large-zip skew seen in real archives).
- **Versioning:** `gold.jsonl` carries `gold_version` (semver); the leaderboard records which version each score used.

## 5. Composite "Gold-Grade Score" + Per-Axis PASS BARS (Gate)

**The composite is informational only and is reported alongside, never instead of, the AND-gate.** A weighted score that gates nothing invites someone to cite a friendly 0.xx while bars fail; it is kept solely as a single-number trend line. **The PASS BARS are the sole gate** — all gating bars must clear simultaneously (AND).

```
GoldGrade(informational) = 0.45·NoMissRate
                         + 0.15·CoverageClean
                         + 0.15·IFAdhere_ThreadStrict
                         + 0.15·TriageRecall_macro
                         + 0.10·LongCtx@EffectiveCeiling
```
(CLOVER-ATTACH, CLOVER-CHRONO, CLOVER-SYNTH are excluded from the composite until they gate; including not-yet-built scorers would inflate a number that means nothing.)

**PASS BARS (the gate):**
1. **NO-MISS (prominent).** `NoMissRate ≥ 99%` of threads have zero misses on the **value-correct** measure, evaluated on `N ≥ 100` threads (§4). Per-class:
   - T1 classes `ref_number`, `date`, `amount` (after canonical normalization), and `party` (from headers): **`value_correct` recall = 100%** on the gold set — these are deterministically anchored, so a miss is a real defect, not a labeling artifact.
   - `attachment` inventory: 100%.
   - `action_item` (semantic, no oracle): **`recall ≥ measured inter-annotator agreement`** for the current `gold_version`, with **every uncovered atom loud-flagged** (zero *silent* action-item misses). A literal 100% is not claimed because the ceiling is annotator agreement; the bar is "match the best a careful human achieves, and never drop silently."
2. **Coverage.** `CoverageClean = 100%`: every gated thread reports `messages_read == messages_total`, `render_failed == 0`, `bytes_truncated == 0`, `attachments_inventoried == attachments_total`. (Attachment-content read is added to this bar only when CLOVER-ATTACH goes live.)
3. **Triage.** Macro per-class recall ≥ 95%; no single class < 90%.
4. **Format / Citation.** `IFAdhere ThreadStrict ≥ 98%`; **100% of emitted action items carry a valid resolvable citation** (verifier-checked).
5. **Long-context (composite path).** `EffectiveThreadCeiling ≥ 128k` tokens at the no-miss floor **via the deployed chunked/iterative-refine path** on operator hardware. If the composite path cannot reach 128k, the bar is the **empirically measured ceiling**, and any thread above it MUST route to chunking (already in `_build_once`) — a thread is never silently fed past the ceiling. The ceiling is a measured number recorded in the leaderboard, not an aspiration; it cannot be quietly waived.
6. **Oracle escape (new).** `oracle_escape_rate == 0` on the current red-team subset, refreshed each gold version. A nonzero rate is a loud failure that seeds new gold atoms before any clearance.

**Future bars (apply only when the feature ships):**
7. **Attachment.** text NED ≤ 0.10; table TEDS ≥ 90; annotation field-F1 ≥ 95 — gates only once attachment ingestion exists.
8. **Synthesis (Phase-4 zero-touch).** issue-clustering F1 ≥ 90; per-thread→project propagation loss = 0; cross-project `NoMissRate ≥ 99%` at the largest bracket the deployed Phase-4 backend demonstrably sustains on operator-controlled hardware.

**Gating statement:** the per-thread pipeline stays manual until bars 1–6 clear on operator hardware (and bar 7 once attachments ship); Phase-4 zero-touch additionally requires bar 8.

## 6. Eval Harness as Code

Lives under `clover/eval/`, invoked by the operator controller / `/dev` panel.

- **`clover/eval/extractors.py` (NEW — prerequisite work item).** The T1 recall-floor extractor: ref-number regex (RFI/NCR/CR/SOI families), date parser→ISO normalizer, amount parser→`{currency,decimal}` normalizer, party-from-header. **Its own recall MUST be characterized against the gold set and recorded before any bar (1, 2) depends on it.** It is explicitly *not* an action-item extractor.
- **Coverage instrumentation in `clover/comprehend.py` (NEW — prerequisite work item).** Add the `coverage` field to the record (lines 415–430); increment `render_failed` on the line-71 except instead of a silent `continue`; record `bytes_truncated` at the 200 KB cap (`estimate_thread_tokens`) and the `max_chars` truncation (`_build_once`); populate `attachments_inventoried` from `_real_attachments`. Bar 2 is unevaluable until this lands.
- **Attachment ingestion (NEW — prerequisite for CLOVER-ATTACH / bar 7).** `_block_text` reads body only; until an attachment reader exists, bars 2 (content) and 7 score vaporware and are explicitly deferred.
- **`clover/eval/gold.jsonl`** — one record per thread/attachment/project: `{id, gold_version, matcher_id, cosine_threshold, normalizer_version, source_ref, required_atoms:[{class,value,canonical,source,ignore,annotators:[...]}], gold_label, gold_tags, gold_timeline, coverage_expected}`.
- **`clover/eval/scorer.py`** — pure scorer: deterministic IFEval-style verifiers; the frozen entailment matcher + canonical date/amount comparison; ACE found-vs-correct split; emits per-suite scores, the AND-gated headline, `critic_recall`, `oracle_escape_rate`, and the literal uncovered-basis miss list. No network; reproducible.
- **`clover/eval/run_eval.py`** — `python -m clover.eval.run_eval --model <id> --gold gold.jsonl`: runs the pipeline over gold inputs, calls the scorer, writes a leaderboard row.
- **`clover/eval/leaderboard.jsonl`** — append-only: `{ts, model_id, config_hash, gold_version, matcher_id, per_suite_scores, composite, bars_passed:bool, ceiling, oracle_escape_rate}`.
- **`clover/eval/history/`** — regression tracking; CI/cron compares each run to last and **fails on any per-class recall regression OR any rise in `oracle_escape_rate`**. Surfaced in `/dev` alongside the model registry and token budgets.

## 7. Recursive Improvement Loop

Run to plateau. Explicit signal → action mapping, with a concrete lever for every failure mode (including when the AI critic itself misses and when no model swap is possible).

1. **Measure** — `run_eval` → leaderboard + miss list + `oracle_escape_rate`.
2. **Identify worst axis** — lowest gating bar relative to its target (recall bars weighted first).
3. **Apply ONE intervention** (signal → action):
   - Missed *catchable* atom (ref/date/amount/party) → **add/strengthen a T1 deterministic extractor** (cheapest, highest reliability).
   - Surface-form date/amount miscount → **extend the canonical normalizer** (new format → canonical value).
   - One pass conflating concerns / low per-pass score → **decompose the pass**.
   - Missed *semantic* atom (implied action item), pass 1 → **add/strengthen the completeness-critic pass**.
   - **Critic itself misses (low `critic_recall`)** → escalate the critic before touching the model: (i) **per-class critic prompts** (a dedicated action-item critic separate from a fact critic); (ii) **self-consistency / ensemble voting on the action-item pass** (sample K times, union the items — recall-first); (iii) **second independent critic pass** with a different decomposition. These are repeatable levers, so "add a critic" is no longer a one-shot.
   - Large gap between schema-strict and content correctness → **fix the prompt's output contract** (not the model).
   - Recall craters past length L (CLOVER-LONGCTX) → **tune chunk size / overlap / iterative-refine depth** in the existing `_build_once`/`_refine` path; route above-ceiling threads to chunking. Only if the composite path is exhausted, **swap to a longer-context local model that fits the operator box**.
   - Triage minority class low → **add learned rules / profile examples** for that class (deterministic `rules.py` wins before AI).
   - **`oracle_escape_rate > 0`** → promote the escaped atoms to gold, then apply the matching lever above for their class (extractor for catchable, critic for semantic).
   - **All deterministic + structural + critic-ensemble levers exhausted and a semantic bar still fails, with no longer-context local model available** → this is a **decision point, not a deadlock**: the product **ships manual-only** for that capability (the human stays in the loop, which is acceptable because the bar's purpose is to prevent *silent* misses, and a manual stage is loud by construction), OR the operator **formally renegotiates the bar** with the new measured ceiling recorded in the leaderboard. An unreachable bar is never silently waived.
4. **Re-measure** on the same gold version.
5. **Keep deltas** only if no axis regressed (guard with `history/` and `oracle_escape_rate`); else revert.
6. **Stop at plateau** — epsilon is **tied to gold-set granularity**: epsilon `≥ 1/N` (and expressed in whole-atom / whole-thread counts, never a sub-quantum percentage). With `N ≥ 100` (bar prerequisite, §4) one thread = 1.0%, so the stop epsilon is **1 thread (≈1%)** across two iterations — never the previous 0.5%, which was finer than a 30-thread grid's 3.3% quantum and would trigger on quantization noise. Prefer deterministic interventions before AI ones throughout.

## 8. Model Selection

The **best LOCAL model is the output of running this suite on operator hardware** (the operator's single multi-tenant local-AI box, ≈128 GB unified-memory class) — not a guess and not an external leaderboard. Cloud models are benchmark-only references (never touch real email). Selection: run `run_eval` for each candidate local model/config; the chosen model **clears all gating PASS BARS at the lowest token/latency cost** with the highest measured `EffectiveThreadCeiling` *on the composite path*. Because the per-thread bar is set against the chunked path (not the raw window), a model that fits the box and holds the composite ceiling is a valid target even though no single-window 128k local model exists. Whole-project Phase-4 synthesis selects separately against the long-context brackets the deployed Phase-4 backend sustains (bar 8). Model choice is a measured, regression-tracked decision recorded in the leaderboard, re-runnable whenever a new local model lands.

## 9. Consolidated-List Bars (CLOVER-LIST-*) — Clover-Native, List-Centric

All six bars score the **consolidated list** per **CASE**, use the **frozen matcher per `gold_version`** (§4) and the **canonical date/amount normalizer** (§3), keep the **ACE found-vs-correct split** (§3 CLOVER-NOMISS), and are **AND-gated per CASE, never averaged over atoms** (§1). Each bar names an **exact measure** and a **pass threshold**. Recall remains the gate; precision/clutter remains a secondary, recoverable penalty (§1) — *except* where a "clutter" failure is shown to be a disguised silent miss, in which case it inherits the hard-zero of the no-miss promise.

### 9.1 CLOVER-LIST-RECALL — List-Level No-Miss *(headline; supersedes §5 bar 1 as the cardinal gate)*
- **Guarantees:** every real-world action that exists *anywhere* in the corpus (any message, thread, channel; including actions implied only by composing ≥2 messages) appears as exactly one live item on the consolidated list. An action mentioned in five places is **one** required item; dropping it once is a miss even if four mentions survived.
- **Exact measure:** per CASE, gold = the consolidated required-action set `G` (each `g` = `{canonical_what, owner, counterparty, due_canonical, terminal_status, source_msg_ids:[≥1, across channels], parent_issue_id}`). An emitted item *covers* `g` iff **exactly one** `s ∈ S` entails `canonical_what` under the frozen matcher AND `s` maps to ≥1 of `g.source_msg_ids`. Report **`found_at_all`** (located) vs **`value_correct`** (located AND owner/counterparty/due/status all correct) separately (ACE). `ListRecall = covered / |G|`. **Headline `ListNoMissRate = (# CASES where value_correct covered == |G| for ALL g) / (# CASES)`.** Sub-metrics: `implied_action_recall` (actions whose evidence spans ≥2 messages), per-class recall on the action's constituent atoms, `propagation_loss = (atoms present in some per-thread comprehension record but absent from the final list) / (atoms present per-thread)`.
- **Pass threshold:** `ListRecall ≥ measured inter-annotator agreement (IAA)` for the current `gold_version` — **with the IAA-laundering guard of §10.3** (IAA is a measured, interrogated input, *not* a ceiling to defer to for hedged/implied classes); AND `ListNoMissRate ≥ 99%` on `N ≥ 100` CASES; AND **every uncovered action is LOUD-flagged (silent list-miss count = 0, non-negotiable)**; AND `propagation_loss == 0` (a consolidation stage that drops an already-extracted atom compares two machine artifacts — any drop is a deterministic bug, not a judgment call, so it is held to hard zero).
- **Why:** a missed obligation is unrecoverable by the user; a spurious one is a one-click dismiss. The bar is IAA, not a literal 100%, because you cannot validly demand the instrument beat the reference it is calibrated against — but IAA is *itself validated* (§10.3) so it cannot launder a shared annotator blind spot. AND-gating per CASE (not an atom average) is the metrology point: a 99% atom average hides the one CASE that dropped the user's single critical action.

### 9.2 CLOVER-LIST-CONSOLIDATE — Dedup / Fragmentation Fidelity
- **Guarantees:** the same real-world action across multiple messages/threads/channels collapses to **exactly one** item — neither fragmented (under-merge: the user sees it twice) nor over-merged (two distinct obligations fused, which **silently hides one inside the other**).
- **Exact measure:** clustering scored against gold clusters (each gold cluster = all source mentions of one real action). (1) `FragmentationRate = (# gold actions split across ≥2 emitted items) / |G|`. (2) `OverMergeCount = (# emitted items covering ≥2 distinct gold clusters)` — reported as a raw count and **each instance is also counted as a CLOVER-LIST-RECALL miss** (the swallowed action is routed into the recall miss list and loud-flagged), so the two bars **cannot be traded off**. (3) `DuplicateRate = (# emitted items mapping to a gold cluster already represented by another emitted item) / |S|`. (4) pairwise-mention precision/recall **reported separately** (not just F1 — the two failure modes have opposite remedies) plus adjusted Rand index. A merge is only valid if owner AND due AND counterparty are **identical** within the item; topical similarity is not sufficient. Report the **deterministic-key share**: of correctly-consolidated items, the fraction joined by a deterministic key (shared `ref_number` / drawing no.) vs by AI similarity — the deterministic share is the trustworthy floor (deterministic-first principle, §2a).
- **Pass threshold:** `OverMergeCount == 0` (hard zero — every over-merge is a disguised silent miss); `FragmentationRate ≤ 1/N` (expressed in whole-action counts, never a sub-quantum %, consistent with §7's epsilon rule); `DuplicateRate ≤ 2%` (pure duplicates are loud and one-click recoverable); pairwise-mention recall `≥ IAA-on-clustering`; **deterministic-key consolidation must cover the reference-bearing cases at 100%** (AI consolidation must never be the path for a case a key could join exactly).
- **Goodhart guard:** because recall rewards one-item-per-action, an optimizer can inflate `ListRecall` by aggressive merging (silently fusing distinct obligations) OR by fragmenting *just under* the `1/N` tolerance. Defenses: `OverMergeCount==0` routed into recall; identical owner/due/counterparty required to merge; the §11 audit **specifically samples merged items**; and `FragmentationRate` carries a **trend alarm** (§9.6 / §11) — creep toward the `1/N` ceiling is treated as a gaming tell, not noise.

### 9.3 CLOVER-LIST-ATTRIB — Attribution Correctness (Who / Whom / Direction / By-When)
- **Guarantees:** each consolidated action is correct on owner (who must act), counterparty (to whom owed), **direction** (the owe-arrow points A→B not B→A), `is_mine` (the user owes it vs is owed it), and due date — including the hard **rebound** case (msg 1: A owes B by Fri; msg 7: reassigned to C, moved to Tue), where the resolved value, not the stale first mention, must win.
- **Exact measure:** on actions `found_at_all` (§9.1), score five fields against gold: `owner_correct`, `counterparty_correct`, `direction_correct`, `is_mine_correct`, `due_correct` (after canonical normalization; **relative deadlines** — "by next Friday", "EOD", "COB tomorrow" — resolved against each message's own sent-timestamp+timezone to ISO **before** comparison; 0-day tolerance — a due date is exact). Report each field separately plus `AttribAll = (# covered where all five correct) / (# covered)`. **Slice by `attribution-stable` vs `attribution-rebound`**; the rebound slice is held to the **same** bar (no allowance for "it was reassigned"). Honest-unknown (`owner=null, needs_review`, loud) is scored as a **partial**, distinct from a wrong-**confident** attribution.
- **Pass threshold:** `owner_correct == 100%`, `direction_correct == 100%`, `is_mine_correct == 100%` (all deterministically anchorable to headers + explicit assignment language — a miss is a real defect; inverting `is_mine` inverts the entire to-do list); `counterparty_correct ≥ 99%`; `due_correct ≥ 99%` (relative-date resolution has genuine timezone/business-day ambiguity); **wrong-confident attribution rate == 0** (asserted a specific owner/due that is wrong, with no flag). Rebound slice held to the stable-slice bar.
- **Why:** a located action with the wrong owner/direction/deadline is worse than a miss — the user acts on bad data with false confidence. Confident-wrong is held to zero; honest-unknown (flagged) is tolerated, converting unrecoverable to recoverable. Identity resolution (§10.2) runs and is scored **before** attribution scoring so a correct attribution that merely uses a different alias of the same person is not penalized.

### 9.4 CLOVER-LIST-STATUS — Status-Resolution (Open vs Done/Superseded)
- **Guarantees:** each action carries the correct **current** terminal status resolved across the full chronology of all its mentions (any channel): requested-then-confirmed-done shows **done**; superseded shows **superseded**; withdrawn shows **cancelled**; done-then-reopened shows **open**. A done item shown open wastes effort (loud); an open item shown done is a **missed alarm**.
- **Exact measure:** gold carries per-action `terminal_status ∈ {open, done, superseded, cancelled, blocked}` and the resolved final due/owner after supersessions. `StatusAccuracy = (# correct terminal_status) / (# covered)`, reported as a **confusion matrix** so the two asymmetric errors are visible: **`FalseClosedRate = (gold-open shown done/superseded/cancelled) / (gold-open)`** (a live obligation vanishes from the active list = silent miss) vs `FalseOpenRate = (gold-closed shown open) / (gold-closed)` (loud clutter). Plus `RebindAccuracy` (final due/owner correct on actions with ≥1 superseding event). **Every asserted closure must cite the resolving message** (resolvable, §9.5 verifier-checked).
- **Pass threshold:** **`FalseClosedRate == 0`** (hard zero — inherits the no-miss promise; on ambiguity the safe default is `open / needs_review`, loud, never a guessed `done`); `StatusAccuracy ≥ 98%`; `RebindAccuracy == 100%` (deterministic once mentions are correctly ordered); `FalseOpenRate ≤ 5%`; 100% of asserted closures carry a valid resolving citation.
- **Why:** the asymmetry is the whole point and a single accuracy number would hide it. Closing an open action silently removes it from what the user acts on — identical in consequence to never extracting it. **Temporal-leak guard:** status gold is built on full chronology, but final-state correctness on a frozen corpus is *not* sufficient — the **streaming** version of this failure is gated separately in §9.6.

### 9.5 CLOVER-LIST-XREF — Cross-Reference Completeness, Resolvability & Provenance
- **Guarantees:** each consolidated action links to **all** its source messages across every channel AND to its parent issue; every link is **resolvable** and points at the **originating** occurrence, not a re-quote.
- **Exact measure:** per covered action with gold source-set `Src(g)` and gold `parent_issue_id`: (1) `SourceSetRecall = mean over covered of |emitted_sources ∩ Src(g)| / |Src(g)|` (linked ALL mentions, not just the first). (2) `ParentLinkAccuracy = (# covered with correct parent_issue_id) / (# covered)`. (3) **`CitationResolvable`** = fraction of emitted `(action, source)` links where a **deterministic verifier** confirms `message_id + char_span` exists, the span **entails** the action, **and the span is the originating occurrence** (`is_origin == true`, `is_quote == false`). A link into a **re-quoted** region is **non-canonical provenance** and counts as a **broken** cross-reference even though the quoted span lexically contains the action — this closes the "resolvable-but-wrong-origin" gameability (stress scenario: quoted-reply). `ChannelCoverage` (source-agnostic sub-metric) = of multi-channel gold actions, the fraction whose emitted source-set spans all channels gold says it appeared on.
- **Pass threshold:** **`CitationResolvable == 100%`** (a citation is a promise of traceability; a broken or wrong-origin one is a lie the user discovers only after spending trust — zero tolerance, machine-checkable); every action carries ≥1 resolvable citation (100%, hard); `SourceSetRecall ≥ 95%` (missing a redundant 4th mention does not cost an action, but the consolidation must be auditable); `ParentLinkAccuracy ≥ 98%`. `ChannelCoverage` gates only once a second channel is live (§12); until then it is exercised structurally (synthetic cases) so the matcher is channel-agnostic **by construction**, never retrofitted — synthetic exercises count only toward harness bring-up, never toward CLEAR (§9.7, §12).
- **Why:** resolvable, origin-correct citation is the user's 2-second verification path and the §11 audit's trace-back path — without it you cannot even diagnose where consolidation dropped something. The de-quoting prerequisite (§6) is what makes `is_origin`/`is_quote` computable; until it ships, XREF is reported-not-gating (§9.7).

### 9.6 CLOVER-LIST-STREAM — Incremental / Temporal Fidelity *(closes the snapshot-vs-stream hole)*
- **Guarantees:** the list is correct **after each new message arrives**, not only as a static snapshot over a frozen corpus. The dominant near-invisible failure is the **consolidation-UPDATE miss**: a new message should **mutate** an existing item (re-open a closed action, change a due date, add a 2nd owner, supersede item #7), but the system instead spawns a fresh duplicate OR silently leaves the stale item unchanged. Snapshot bars (§9.1–§9.5 and all of §1–§5) are **structurally blind** to this because they score one final extraction; a raw-source auditor working from the frozen corpus is **equally blind**. This bar is therefore mandatory and gating, not a footnote.
- **Exact measure:** **replay each CASE message-by-message in true chronological order**; after each arrival, diff the system's list-**mutation** against the gold list-mutation. Gate three rates: `UpdateRecall = (required mutations applied) / (required mutations)`; **`DuplicateOnUpdateRate = (new item spawned when an existing item should have been edited) / (required mutations)`**; **`StaleClosedRate = (a superseding/reopening message arrived but the prior item was neither updated nor flagged) / (such events)`**. Reuse the existing staleness primitive — `is_stale` / `thread_sig` keyed on source `{n, end}` (`comprehend.py:130–143`) — **lifted from the thread to the ACTION**: an action whose source-set `{n, end}` changed is `is_stale=true` and must be re-resolved or loud-flagged ("brown clover" at the action level).
- **Pass threshold:** `UpdateRecall ≥ IAA`; **`DuplicateOnUpdateRate == 0` (hard)**; **`StaleClosedRate == 0` (hard)** — both are streaming forms of the silent miss (a dropped/duplicated live obligation) and inherit the no-miss hard zero. The status whipsaw (done → three-weeks-later reopened) must surface the reopened obligation with its new due date on the next replay step, or it fails here even if the *snapshot* terminal status was correct.
- **Why:** the user lives in a stream; certifying a perfect static list while the incremental path quietly accumulates duplicate/stale actions is the single thing most likely to make a high-scoring system untrustworthy in daily use. Without a streaming replay harness the §11 fidelity audit cannot see this class **at all**.

### 9.7 CLOVER-LIST-ACTIONABILITY — List-Level Signal-to-Noise *(human-attention miss)*
- **Guarantees:** the list as a *whole*, under finite attention, stays trustworthy enough that the user reads every item. The real harm of fragmentation/noise is not the dupes themselves: a noisy list makes the user **stop reading carefully**, at which point a perfectly-extracted real action is **skimmed past and missed in the human** — a miss downstream of a 100%-recall score that the per-item bars cannot model. Promoted from a blind-spot paragraph to a **first-class gate**.
- **Exact measure:** `SpuriousItemRate = (# emitted items that map to no gold action) / |S|`; `UrgencyOrderConcordance` = rank correlation between emitted order and gold urgency (overdue/soonest-due first); plus a **consolidation-completeness** check (closes the merge-attribute-loss hole, stress scenario: union-loss): the merged item's attribute set must be the **union** of its source mentions' constraints (most-binding due date, all owners, all constraints) — `AttributeUnionLoss = (# merged items that dropped a constraint any source mention carried) / (# merged items)`, scored distinctly from recall and dedup (every mention WAS extracted; the loss happens at the merge step).
- **Pass threshold:** `SpuriousItemRate ≤ 10%` (trended down; the gate is conservative because the harm is non-linear in noise); **`AttributeUnionLoss == 0` (hard — a lost deadline on a correctly-merged item is a silent value miss)**; `UrgencyOrderConcordance` reported and trended (not a hard gate while ordering policy is still being tuned).

### 9.8 CLOVER-SRCAGNOSTIC — Source-Agnostic Channel Invariance
- **Guarantees:** list quality is invariant to which channel an action arrived on, and survives **cross-channel** consolidation (an email "please issue the cert" + a WhatsApp "cert done by Fri" = one action). Makes the codebase's pluggable `MailSource` design (`clover/sources/base.py`; dedup on `(folder, validity, key)`) **measured, not assumed**.
- **Exact measure:** run the full §9 battery on three slices — email-only; each added channel in isolation; the mixed multi-channel corpus. Report `PerChannelListRecall` per slice; `ChannelParityGap = max(PerChannelListRecall) − min(PerChannelListRecall)`; `CrossChannelMergeRate = (# cross-channel gold actions correctly collapsed to one) / (# such gold actions)`; `CrossChannelConsolidation` = FragmentationRate/OverMergeCount measured **only** on gold clusters whose mentions span ≥2 channels (the hardest dedup case).
- **Pass threshold:** `ChannelParityGap ≤ 1 action-equivalent`; on cross-channel clusters `OverMergeCount == 0` AND `FragmentationRate == 0`; a NEW channel cannot enter the mixed gate until its isolated slice clears LIST-RECALL and LIST-ATTRIB at the email bars; **adding a channel may not regress any existing channel's `PerChannelListRecall` (hard regression guard)**.
- **Critical scope limit (closes the synthetic-fixture false-confidence hole):** **synthetic** cross-channel fixtures validate the *scorer's plumbing only* and count **solely** toward Stage-0 harness bring-up — **never** toward CLEAR. A high `CrossChannelMergeRate` on synthetic data is the most dangerous false-confidence in the design, because the fixture author already knows the answer and gives both renderings matching identity hints, masking the real identity-resolution gap (email alias vs phone handle vs display-name variant). Real channel onboarding rides the §12 ladder with **real** chat data and **real** adversarial identity gold; see §12.

## 10. CASE-Level Gold (extends §4; does not duplicate it)

§4's 3-tier evidence model, frozen matcher, double annotation, `oracle_escape_rate`, `N ≥ 100`, growth rule, and stratification **all carry forward unchanged** to the CASE level. The additions below are the deltas the consolidated list forces.

### 10.1 The CASE unit and the existence/value escape split *(closes the unverifiable-denominator hole)*
The gold unit is the **CASE** (a real situation spanning *N* messages / *M* channels), built by the §4 protocol (deterministic bootstrap → double annotation → red-team). The cardinal extension is to **split `oracle_escape_rate` (§4) into two first-class, separately-gated metrics**:

- **`oracle_escape_rate_EXISTENCE`** = actions the auditor found that gold **never contained at all** (the obligation class no human in the loop noticed). This is the *denominator* of `ListRecall` — it is what makes the denominator verifiable.
- **`oracle_escape_rate_VALUES`** = wrong attributes (owner/due/status/source/merge) on actions gold **already** contained.

**`oracle_escape_rate_EXISTENCE` is gated to HARD ZERO across gold versions**, separately from the IAA-bounded value bars. This is the **only** structural defense against the named blind spot: action-EXISTENCE has **no deterministic floor** (verified — `_verify_facts` anchors only refs/dates/amounts/parties, never the existence of an obligation), so `ListRecall = covered/|G|` measures recall against "what humans already knew to look for" unless existence-escape is independently driven to zero. A `ListNoMissRate` of 100% with `oracle_escape_rate_EXISTENCE > 0` means **only** "we found everything we were already capable of describing as findable" — precisely the high-score-that-still-misses the objective names as proof the **benchmark**, not the model, is wrong.

### 10.2 Identity resolution is scored before attribution/consolidation
Owner/counterparty matching and cross-channel dedup depend on resolving the same person/action across email aliases, display-name variants, and (later) chat handles. The **identity-resolution layer is a scored artifact in its own right** and is computed **before** §9.3/§9.2 scoring, so correct attribution that merely uses a different alias is not penalized and false-fragmentation from weak identity resolution is attributed to the right cause. For a new channel, identity-resolution gold is **adversarial** (the same person under deliberately mismatched alias/handle/display-name) and must clear before the channel joins the mixed corpus (§12).

### 10.3 IAA is interrogated, not deferred to *(closes the IAA-laundering hole)*
Setting recall at "inter-annotator agreement" is honest only if IAA is not itself laundering a **correlated** blind spot — two annotators who *agree* a hedged/implied action "is not an action" produce a high IAA that legitimizes a real miss (agreement ≠ correctness). Therefore:

- Maintain a **curated adversarial existence set** — politeness/indirect ("it would be great if…", "kindly advise", "when you get a chance"), interrogative-as-request ("are we still waiting on the cert?"), negation-flip, implied-only (an obligation inferable only from composing ≥2 non-actionable facts), consolidation-emergent, attachment-borne, table-bound — annotated by the **third existence-hunter** of §11.
- **Measure annotator recall against this adversarial set first** to *characterize* (not assume) the shared blind spot before freezing IAA, and an explicit annotation guideline: **any deadline-bearing or beneficiary-bearing soft/indirect/interrogative request IS an action.**
- **Decouple the existence-recall reference from IAA** for these hedged/implied classes: existence is gated by `oracle_escape_rate_EXISTENCE == 0` on the adversarial set (§10.1), independent of the IAA-bounded value bars.

### 10.4 Gold must be built from RAW / rendered source, never the flattened fixture *(closes attachment & table holes)*
Verified pipeline blind spots: comprehension reads **body text only** (`_thread_messages` → `render_message`; attachment content never extracted; `.eml` capped at 200 KB), strips **all** HTML via `_TAG` (tables flatten to token soup, destroying column bindings), and never de-quotes. **If gold is built from the same rendered-thread fixture the pipeline consumes, the annotator inherits the pipeline's exact blind spots and the benchmark certifies them.** Therefore:

- **Gold for attachment-bearing CASES is built from the opened attachment**; gold for **table-bearing** messages is built from the **rendered HTML table** (column bindings intact); gold provenance for **quoted** actions records the **originating** message, not the re-quote.
- This is a gold-construction rule, separate from the pipeline fixes in §6; gold leads the pipeline so the bar can *detect* the miss even before the pipeline can avoid it.

### 10.5 Planted-defect positive controls *(validate-the-validator's sensitivity floor)*
Inject synthetic CASES with a **known** defect; the scorer **must** fail every one. `benchmark_sensitivity = (planted defects caught) / (planted defects)`; `< 100%` means the matcher/threshold/scorer is blind to a defect class — **the validator is broken before the model is even in question**. Mandatory, broad coverage: (a) a dropped attachment-action; (b) a cross-thread/cross-channel two-mention action that MUST merge; (c) a multi-row-table cross-row mis-binding; (d) a cited-the-re-quote-not-the-origin provenance defect; (e) a re-opened-after-done status whipsaw (snapshot AND streaming); (f) an over-merge fusing two distinct obligations; (g) an attribute-union-loss merge (most-binding deadline dropped); (h) near-miss paraphrases the matcher MUST **reject** (guards matcher false-cover, §4 / Open Q5). `benchmark_sensitivity == 100%` on every class is a precondition for any CLEAR.

## 11. Benchmark Fidelity: False-Clear Rate & Polish-the-Benchmark-First *(validate-the-validator)*

The measurement bar is the **sole lightpole** for recursive optimization (§7). If the lightpole points at a clear road that has a pothole, you fix the lightpole **first**. This section makes the benchmark's own correctness a **first-class, gated** property.

### 11.1 The metric: False-Clear Rate (FCR)
**`FCR = (# CASES the benchmark CLEARED as zero-miss that a deeper independent authority later proved contained ≥1 real defect) / (# CASES cleared)`**, where a *defect* is the union of all list-level failure classes: a dropped action, an over-merge that swallowed an action, a wrong/reversed attribution, a false-closed status, a missing/wrong-origin source link, an attribute-union-loss, or a streaming duplicate/stale-closed. FCR measures the **instrument**, not the model: a model can be perfect and FCR still nonzero if the gold/matcher/taxonomy is blind to a real miss class. **`FCR = 1 − PPV(PASS)`** — one minus the positive predictive value of a PASS. `FCR > 0` means the **benchmark** is wrong (Goodhart realized), and per the ordering rule (§11.3) it is fixed **before** the model is touched.

### 11.2 How FCR is measured — three independent estimators (the benchmark cannot grade its own homework)
1. **Adversarial existence audit (extends §4's red-team to the CASE + existence split).** For a rotating sample of CLEARED CASES, auditors **exhaustively re-derive the consolidated action set from RAW source across all channels — opening every attachment, rendering every table, de-quoting every reply** — with **no sight** of gold, bootstrap candidates, or system output. Two distinct auditor roles:
   - the standard red-team auditor (value escapes → `oracle_escape_rate_VALUES`);
   - **a THIRD, demonstrably out-of-distribution existence-hunter** whose **sole** brief is to find actions gold itself omits (implied, politeness, interrogative, quoted-origin, attachment-borne, table-bound, cross-thread/cross-channel-emergent) → `oracle_escape_rate_EXISTENCE`. This auditor must be from a different pool, never primed with gold/guidelines/output, working from raw source — **if the auditor shares the annotator pipeline, oracle-escape is circular and reads a false zero** (stress scenario: gold-fatigue miss). Each escape is tagged with its **atom class**.
2. **Planted-defect positive control (§10.5).** `benchmark_sensitivity` is a continuous estimator of scorer blindness; any class `< 100%` is a standing FCR risk.
3. **Field-truth / production ledger.** Every live user correction on a previously-CLEARED CASE ("this was missed / wrong owner / already done / [user adds an action by hand]") is logged with timestamp and atom class. This is the **only** estimator grounded in reality and is the **final arbiter** — but it is **biased and lagging for the worst class**: users only correct misses they *notice*; the never-noticed action generates no signal. **Its zero is therefore never sufficient on its own**; existence-escape (estimator 1, role 3) is the required complement for the invisible class.

**Statistical gating.** FCR is computed per `gold_version` over all three estimators. Report the **Clopper-Pearson / Wilson UPPER bound** on FCR (not the point estimate) as the gate — `FCR = 0` on a handful of cleared CASES is near-meaningless. Size the audit sample so the upper bound can reach a pre-registered target; weight 100% of user-reported misses (free, real ground truth) into FCR. Calibration framing: the benchmark is an instrument, auditor+user are the trusted reference standard, FCR is the instrument's measured error, and the audit cadence is the recalibration interval (§ Open Q6).

### 11.3 The Polish-the-Benchmark-First loop (hard ordering rule, prepended to §7)
On **ANY** pass-but-real-miss event from **any** estimator, the loop touches the **benchmark, never the model**, until FCR returns to a confidence-bounded zero. This becomes the **first gate of §7**, *before* §7 step 2 ("identify worst axis"):

1. **FREEZE** model/prompt/chunking optimization **and freeze scale-up** (§12). No model-tuning commit may land while `FCR_upper > 0`.
2. **CLASSIFY** the escaped defect into an existing list-level class, or open a **NEW atom class** (e.g. "interrogative-as-action", "action raised in chat closed in email", "deadline relative-to-an-event", "consolidation-emergent across channels").
3. **PROMOTE** the CASE + missed atom into gold as a permanent regression seed (full source-set + canonical resolution + originating provenance), and add ≥3 hand-mined siblings. New class → add it to the gold taxonomy, the `action` schema if needed, the §10.3 adversarial existence set, and the §10.5 planted-control set.
4. **DIAGNOSE WHY the benchmark cleared it and REPAIR the instrument:**
   - **matcher false-cover** (threshold too loose; a wrong/misattributed item entailed gold) → retune against the ACE found-vs-correct split, re-freeze threshold, bump `gold_version`;
   - **gold blind spot** (auditor-only atom) → promote to gold; characterize the bootstrap extractor's recall on that class;
   - **scorer can't express the defect** (e.g. no over-merge/union-loss/streaming metric for that pattern) → extend the scorer;
   - **protocol gap** (single-annotated class that needs double / needs the existence-hunter) → upgrade the annotation protocol;
   - **threshold gap** (a bar's threshold was too lenient for that class) → tighten with the new measured ceiling recorded.
5. **RE-RUN** the planted-defect control to confirm `benchmark_sensitivity == 100%` for that class, then **re-run the FULL suite on the SAME model/config** to confirm the CASE now **FAILS** — this is the regression test *for the benchmark*. Also **re-score the historical leaderboard**: a previously-"passing" model may now correctly FAIL (a Goodhart unmasking — that model was never actually passing).
6. **Only after `FCR_upper` is back under target on the refreshed gold may model optimization resume.**

**Logged invariants.** Benchmark-polish commits and model-tuning commits are **separate** and recorded in the leaderboard with the `gold_version` bump (so "the score went up" is never confounded with "the benchmark got weaker"). A score is comparable to a prior score **only** at the same `gold_version` AND with `FCR == 0` throughout; a model improvement claimed across a `gold_version` bump where FCR was nonzero in between is **rejected as un-validated**. `oracle_escape_rate_EXISTENCE`, `oracle_escape_rate_VALUES`, FCR, and `benchmark_sensitivity` are **gating bars with the same zero-tolerance standing as recall** — a rise in any fails CI exactly like a recall regression (extends §6 `history/`). The §7 stop condition is **doubly gated**: plateau on the bars **AND** FCR held at confidence-bounded zero across the last *K* gold versions (validator drift counts as non-convergence even if the headline is flat).

### 11.4 Why a passing score now provably predicts zero real misses at the trialed scale
A PASS is trustworthy **iff** `FCR_upper ≤ target` on independent reference channels that do **not** share the gold's blind spots. The three holes that previously made a PASS hollow are each closed by a *gating* mechanism, not a footnote: existence-blindness → `oracle_escape_rate_EXISTENCE == 0` via the out-of-distribution third auditor (§10.1, §11.2); snapshot-blindness → CLOVER-LIST-STREAM (§9.6) replay; fixture-blindness → gold built from raw/rendered source (§10.4) + the refusal to CLEAR a corpus with unparsed attachments/tables (§9.7 / §11.5). What remains genuinely irreducible is named honestly in **Residual Risks** rather than papered over.

### 11.5 Hard CLEAR preconditions (what may never be cleared)
- **No zero-miss CLEAR on a corpus containing unparsed attachments or structure-destroyed tables.** Until attachment-extraction and table-preservation ship (§6), ship a **deterministic T1 loud-flag** — "this CASE has an unread actionable attachment (xlsx/pdf/docx)" / "this message has a flattened table" — so the miss is at least **LOUD (recoverable)**, never silent. Attachment-borne and table-bound action recall is **REPORTED-NOT-GATING** until those stages exist (consistent with §3's CLOVER-ATTACH posture).
- **All §9 list-level bars are REPORTED-NOT-GATING until the `action` object + real consolidation pass + de-quoting/attachment/table extraction exist** (§1.5, §6). Today they score vaporware; reporting them is fine, *clearing* on them is forbidden.
- **Synthetic cross-channel fixtures never count toward CLEAR** (§9.8) — only toward Stage-0 bring-up.

## 12. Staged Real-Case Trial: 1 → ~10 → ~100 → ~1000+ *(gated by FCR == 0 at the prior stage)*

A clinical-trial-style escalator where the **patient is the BENCHMARK earning trust**, not the model. Stages are sized in **messages** (source-agnostic count across channels); each level is a superset of the prior. **Hard invariant, enforced in `run_eval` (not merely warned): NEVER scale up while `FCR_upper > 0` at the current stage.** Each rung re-runs **all** prior rungs' gold (no regression, no silent un-clearing); the gold set **grows** at every stage from escapes + production corrections, so a later rung is always scored against a **stricter** benchmark (trust ratchets up, never down).

- **STAGE 0 — Instrument bring-up (n=1 CASE, ~1–5 messages).** Hand-build the consolidated gold for one rich CASE (multiple actions, one superseded/re-bound action, one cross-message duplicate, one implied-only action, one quoted-origin action, one table-bound action). Run the suite. Purpose is **not** model quality — it is to prove the harness, the new `action` schema, the matcher, the verifier, the **streaming replay** (§9.6), and the scorer all execute end-to-end and the scorer reproduces hand-computation **bit-for-bit**. Synthetic cross-channel and synthetic table/attachment fixtures are exercised here for plumbing only. **Exit:** scorer == hand-scored; `benchmark_sensitivity == 100%` on all planted classes (§10.5); zero **unclassified** escapes; FCR audit of the 1 CASE = 0.

- **STAGE 1 — Calibration (n≈10 CASES, ~50–150 messages; ≥2 channel-split pairs once chat lands).** Double-annotate; **measure IAA and interrogate it against the adversarial existence set (§10.3)** — this SETS the recall/attrib ceilings and characterizes the shared blind spot; tune and **FREEZE** the matcher cosine threshold against the ACE split; freeze `gold_version 1`. Run the **full adversarial audit on 100% of cleared CASES** (cheap and maximally informative at this n), including the third existence-hunter. **Exit:** all gating bars defined with thresholds tied to measured IAA; `oracle_escape_rate_EXISTENCE == 0` and `oracle_escape_rate_VALUES == 0` across all 10; **no NEW atom class for a full clean pass**; FCR = 0 with its upper bound reported. **Do NOT report any bar as "cleared" here** — n is below the §4 `N ≥ 100` rule; advancement is on "no new escape class", not on a small-sample zero.

- **STAGE 2 — First real gate (n≈100 CASES, ~1000–5000 messages).** The smallest n at which a 99% `ListNoMissRate` is statistically meaningful (1 CASE = 1%). **Stratify** to mirror the real archive distribution (§4: thread length, attachment-heavy, forwarded/quoted clutter, cross-channel, re-binding) and **over-sample rebound and cross-thread cases** relative to natural frequency (they are where false-closed and stale-attribution silent misses live; reported as a separate slice held to the same bar). Full scorer run gates the §9 bars (AND-gated). Audit a rotating **≥20%** of cleared CASES + **100% of narrowly-passed** CASES (within one atom of failing — where false-clears hide) + 100% of user-flagged. **Exit to Stage 3 ONLY IF:** all §9 bars clear simultaneously AND `FCR` point estimate = 0 AND **`FCR_upper ≤ pre-registered target (e.g. ≤1%)`** AND `oracle_escape_rate_EXISTENCE == 0` AND `oracle_escape_rate_VALUES == 0` AND `benchmark_sensitivity == 100%` AND no per-class recall regression vs Stage 1 (`history/`). **Only at Stage-2 clearance may the per-CASE pipeline flip from manual to zero-touch auto-run.** If any false-clear fires: run the §11.3 polish loop, re-freeze `gold_version`, and **RESTART the stage's audit clock** — you do not advance on a benchmark that just proved itself blind.

- **STAGE 3 — Sustained zero-touch (n≈1000+ CASES, the full backlog).** Scale to the full archive **only after** Stage 2's `FCR_upper` holds. The mass-run / zero-touch auto-pipeline flips on here, but **trust is continuously re-earned**: audit cadence becomes a continuous rotating sample sized to keep `FCR_upper` under target per `gold_version`; 100% of user corrections still feed FCR. A **single confirmed false-clear** (especially a user-reported real miss on a cleared CASE) trips a **circuit breaker**: auto-run reverts to **manual (loud)** for the affected **stratum**, the polish loop runs, `gold_version` bumps, and the stratum must **re-clear** (re-run Stage 2 → 3 for it) before auto-run resumes. Already-shipped lists are not auto-rolled-back, but further scale-up is frozen until `FCR_upper` returns under target.

- **CHANNEL ONBOARDING rides this same ladder (closes the cross-channel-retrofit hole).** A NEW channel (WhatsApp, etc.) re-enters at its own **mini Stage 0/1 on REAL chat data** with **REAL adversarial identity-resolution gold** (alias/handle/display-name mismatches of the same person), gated like a new model candidate, with the identity-resolution layer (§10.2) **built and independently scored before the channel joins the mixed corpus**. Non-email channels have **no Message-ID/References threading and no headers for party attribution** — the deterministic anchors the entire T1 floor depends on **evaporate** — so source-agnosticism must be designed into the gold schema and matcher **now** (via the structural synthetic exercises of §9.8), and validated with **real** data at onboarding. Synthetic cross-channel scores **never** count toward clearance. Hard regression guard: adding a channel may not lower any existing channel's recall.

**Cross-stage ratchets (extend §4 growth rule + §6 `history/`):** (1) the **regression ratchet** — gold only grows; every production miss, every user `Resolve`, every red-team/existence escape is promoted as a permanent seed; per-class recall may never regress vs `history/`. (2) the **trust ratchet** — you may only be at a stage if every lower stage still passes on the current `gold_version`. (3) the **overfit alarm** — keep a held-out, never-optimized-against red-team partition (rotated); track the score gap between optimized-gold and held-out gold as an overfit signal, since promoting every production miss increasingly encodes one model's error signature. Every scale-up is recorded in `leaderboard.jsonl` as an auditable decision `{stage, n_cases, n_messages, gold_version, per-bar scores, oracle_escape_rate_EXISTENCE, oracle_escape_rate_VALUES, FCR + CI, benchmark_sensitivity, bars_passed}`.
